# quoridor_alphazero

AlphaZero-style move engine for 5Г—5 Quoridor. ROS2 package, drop-in
wire-compatible with `quoridor_ai_move` (same `/quoridor/compute_move_request`
and `/quoridor/compute_move_response` topics).

This package replaces the DQN approach in `quoridor_ai_move` with the
algorithm AlphaGo Zero / AlphaZero use: a single neural network with two
heads (policy + value), trained on data generated by Monte Carlo Tree
Search using that same network as the leaf evaluator.

---

## Why this instead of DQN?

DQN was designed for single-agent environments. In a two-player perfect-
information game, the Bellman bootstrap `r + ОіВ·max_a Q(s', a)` conflates
"my move quality" with "my opponent's reply quality" because `s'` is the
state *after* the opponent has moved. This produces well-known pathologies
in self-play: mutual stalling, oscillating policies, brittle learning.

AlphaZero handles two players natively:

- The **value head** is trained against the actual game outcome `z вҲҲ {-1, 0, +1}`
  (no bootstrap, no temporal-difference noise).
- The **policy head** is trained against the **MCTS visit-count distribution**
  from each position вҖ” a strictly stronger signal than the network's own
  best guess, because MCTS gets to look ahead using the network many
  times before committing to a move.
- **Two-player minimax backup** in MCTS: when a child node's value is
  backed up to its parent, the sign is flipped (opponent's good is my bad).

Net effect: the network bootstraps off a search that is itself guided by
the network вҖ” a virtuous circle that doesn't have the self-play
pathologies DQN does.

### One self-play game (`self_play.py`)

```python
while game not over:
    counts = mcts.run(board)               # 64 simulations from current state
    ПҖ = visit_counts_to_policy(counts, П„)  # П„=1 first 12 plies, П„=0 thereafter
    record (state, ПҖ, side)
    sample action ~ ПҖ
    board.apply_move(...)

z = +1 / -1 / 0 from each side's POV (filled in after game ends)
push every (state, ПҖ, z) into ReplayBuffer
```

### One MCTS simulation (`mcts.py`)

From the root, repeatedly:

1. **Select** child action `a` maximizing `PUCT = -Q(child) + c_puct В· P(s,a) В· вҲҡ(ОЈN) / (1 + N(s,a))`.
   The minus on `Q` is the two-player flip вҖ” what's good for the child is bad for the parent.
2. **Apply** the move on a cloned board (the real board is never mutated).
3. **At a leaf:** call the network once on the leaf state. Mask illegal
   actions, renormalize policy, get value `v`.
4. **Back up** along the visited path. At each step up the tree, flip
   the sign of the value before adding it to that node's running sum.

After all simulations, the root's child visit counts are the move-selection
distribution. Counts are also the *training target* for the policy head вҖ”
"play in proportion to how much MCTS thought about each move."

### Training (`train.py`)

Each game contributes вүҲ `ply_count` samples. After each game we run a
small number of Adam steps (`--updates-per-game`, default 8) on a random
batch from the buffer. The buffer is circular, persisted to disk on
every `--save-every` games, and reloaded on `--resume`.

```
loss = вҲ’ ОЈ ПҖ_mcts В· log softmax(p_logits)     # cross-entropy policy loss
       + (z вҲ’ tanh(v_logit))ВІ                  # MSE value loss
       + L2(weights)                           # built into layer kernels
```

## Files

| File | What it does |
|---|---|
| `encoder.py` | Board вҶ’ `(5, 5, 7)` tensor in agent-POV coordinates; flat 56-action index вҶ” `Move` object; legal-action mask. Same channel layout / action mapping as `quoridor_ai_move`. |
| `network.py` | Dual-head ResNet. Conv stem вҶ’ N residual blocks (default 32 filters Г— 3 blocks, вүҲ 80k params) вҶ’ policy head (logits over 56 actions) + value head (scalar `tanh вҲҲ [-1, 1]`). Smaller than the DQN net because the 5Г—5 state space doesn't need more capacity. |
| `mcts.py` | PUCT MCTS. Mask-aware priors, two-player minimax backup, optional Dirichlet noise at the root for self-play exploration. Uses `QuoridorBoard.copy()` to simulate without mutating the real board. |
| `self_play.py` | One full game with MCTS for both sides. Returns `[(state, ПҖ_mcts, side)]` plus the outcome status; `z` is filled in for every sample once the game ends. |
| `replay_buffer.py` | Fixed-capacity circular buffer of `(s, ПҖ, z)` triples. Allocation-free sampling. Pickle save/load so a long run survives restarts. |
| `train.py` | The training loop above. Atomic save with 3-deep rotating backup (`az_net.weights.bak1.h5` вҖҰ `bak3.h5`) so a bad run can be rolled back. |
| `ai_move_node.py` | ROS2 inference node. Subscribes to `/quoridor/compute_move_request`, runs MCTS with the trained network, publishes the chosen move. Wire-compatible with `quoridor_ai_move/ai_move_node`. |
| `launch/play_vs_az.launch.py` | Brings up `state_manager`, `web_interface`, and the AZ inference node. |

## Usage

### Build
```bash
cd /rs2_ws
colcon build --packages-select quoridor_alphazero --symlink-install
source install/setup.bash
```

### Train (fresh)
```bash
python3 -m quoridor_alphazero.train \
    --episodes 5000 --simulations 64 --updates-per-game 8 \
    --model-dir /rs2_ws/quoridor_az_models/run1 \
    --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1
```

### Train (resume)
Auto-loads weights, replay buffer, and switches to `--lr-resume` (2e-4):
```bash
python3 -m quoridor_alphazero.train --resume --episodes 5000 --model-dir /rs2_ws/quoridor_az_models/run1 --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1_2 --batch-size 1024 --lr-resume 8e-4

python3 -m quoridor_alphazero.train --resume --episodes 5000 --model-dir /rs2_ws/quoridor_az_models/run1 --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1_3 --batch-size 1024 --lr-resume 8e-4 --workers 12

```

### Play against it
You play through the web UI, the AZ engine plays bot. Inference can use
much heavier search than training вҖ” same weights, no retraining:
```bash
ros2 launch quoridor_alphazero play_vs_az.launch.py model_dir:=/rs2_ws/quoridor_az_models/run1 simulations:=400
```

### Recover from a bad checkpoint
Every save rotates the previous weights to `bak1.h5`, etc. To roll back:
```bash
cp /rs2_ws/quoridor_az_models/run1/az_net.weights.bak1.h5 \
   /rs2_ws/quoridor_az_models/run1/az_net.weights.h5
```

## Key hyperparameters

| Flag | Default | What it controls |
|---|---|---|
| `--simulations` | 64 | MCTS sims per move during self-play. More = stronger data but slower. At inference (ROS node) push to 200вҖ“400. |
| `--c-puct` | 1.5 | PUCT exploration weight. Higher = more exploration of low-prior moves. |
| `--dirichlet-eps` | 0.25 | Fraction of Dirichlet noise mixed into root priors during self-play (0 at inference). |
| `--temp-moves` | 12 | Plies of high-temperature (П„=1) move selection at the start of each game. After this point, plays argmax for sharper data. |
| `--updates-per-game` | 8 | Gradient steps after each self-play game. Bounded вҖ” can't run away on long games. |
| `--lr` / `--lr-resume` | 1e-3 / 2e-4 | Learning rate; `lr-resume` is used automatically when `--resume` is set, to avoid Adam overwriting prior weights. |
| `--filters` / `--blocks` | 32 / 3 | Network width / depth. Defaults are intentionally smaller than the DQN net вҖ” 5Г—5 doesn't need more. |

## TensorBoard signals to watch

| Tag | Healthy range | What it tells you |
|---|---|---|
| `loss/policy` | drops 4 вҶ’ 2.5 | Network is starting to agree with MCTS. Stuck near 4 (= ln 56) means uniform predictions, search isn't biting yet. |
| `loss/value` | drops 0.5 вҶ’ 0.1 | Network is learning to predict outcomes. Rising = something is overfitting or the buffer is dominated by stale games. |
| `game/avg_length_50` | 15вҖ“35 | Decisive games. Pinned at `--max-plies` (75) means search is too weak to find a winning line вҖ” increase `--simulations`. |
| `game/bot_win_rate_50` | near 0.5 | In self-play, both seats use the same network. A persistent gap means there's a first-move advantage the policy can't yet equalize. |
| `game/decisive_rate_50` | вҶ’ 1.0 | Fraction of recent games that didn't time out. Should approach 1 once the network can finish games. |
| `loss/total` | declining | Sum of policy + value + L2. The headline number. |

## Practical tips

1. **Start small.** First validate the loop with `--simulations 16 --episodes 100`. If `loss/policy` is dropping, scale up. If it isn't, something is wrong with encoding/MCTS.
2. **Search > network capacity.** If results are weak, double `--simulations` before doubling `--filters`. Stronger search produces better training data, which trains the same net to play better.
3. **Inference search вү« training search.** Train at 64 sims/move, play at 400+. Same model, much stronger play.
4. **Don't mix DQN and AZ checkpoints.** They share the encoder but the network architectures and weight filenames differ (`qnet.weights.h5` vs `az_net.weights.h5`). Use separate `model_dir`s.
5. **Save the buffer.** A populated `replay_buffer.pkl` is worth as much as the weights вҖ” it represents thousands of MCTS simulations. Don't `rm -rf` the model dir.

## What's intentionally absent vs `quoridor_ai_move`

| Feature in DQN package | Absent here, becauseвҖҰ |
|---|---|
| Оө-greedy exploration | MCTS + root Dirichlet noise replaces it. |
| Reward shaping (PBRS) | The value target is the actual game outcome вҖ” no shaping needed. |
| Step penalty | Same вҖ” outcome-only training. Stalling is naturally penalized: a long unfinished game gives `z = 0` to both sides, so the network learns there's no reward in dragging things out. |
| Online learning in the ROS node | AZ learning needs full self-play games at training-time scale. Per-move callbacks aren't the right granularity. Train offline with `train.py`, deploy the weights, repeat. |
| `train_vs_minimax` curriculum | Not needed: the network bootstraps from MCTS using its own (initially random) value estimates, which is enough signal to escape the cold start. If you want a curriculum, the right way is to use the minimax engine as the *opponent* in `play_game` вҖ” easy to add later. |
