# quoridor_alphazero

AlphaZero-style move engine for 5×5 Quoridor. ROS2 package, drop-in
wire-compatible with `quoridor_ai_move` (same `/quoridor/compute_move_request`
and `/quoridor/compute_move_response` topics).

This package replaces the DQN approach in `quoridor_ai_move` with the
algorithm AlphaGo Zero / AlphaZero use: a single neural network with two
heads (policy + value), trained on data generated by Monte Carlo Tree
Search using that same network as the leaf evaluator.

---

## Why this instead of DQN?

DQN was designed for single-agent environments. In a two-player perfect-
information game, the Bellman bootstrap `r + γ·max_a Q(s', a)` conflates
"my move quality" with "my opponent's reply quality" because `s'` is the
state *after* the opponent has moved. This produces well-known pathologies
in self-play: mutual stalling, oscillating policies, brittle learning.

AlphaZero handles two players natively:

- The **value head** is trained against the actual game outcome `z ∈ {-1, 0, +1}`
  (no bootstrap, no temporal-difference noise).
- The **policy head** is trained against the **MCTS visit-count distribution**
  from each position — a strictly stronger signal than the network's own
  best guess, because MCTS gets to look ahead using the network many
  times before committing to a move.
- **Two-player minimax backup** in MCTS: when a child node's value is
  backed up to its parent, the sign is flipped (opponent's good is my bad).

Net effect: the network bootstraps off a search that is itself guided by
the network — a virtuous circle that doesn't have the self-play
pathologies DQN does.

## How it works

```
                                         ┌───────────────────┐
                                         │  ReplayBuffer     │
                                         │  (s, π_mcts, z)   │
                                         └────────▲──────────┘
                                                  │ append
                          ┌───────────────────────┴────────────┐
                          │                                    │
            run MCTS      │                                    │  random batch
            from each     │                                    │
            position      │                                    ▼
   ┌────────────────┐     │                          ┌──────────────────┐
   │  Self-play     │─────┘                          │  Train step      │
   │  one full game │                                │  loss = CE(π‖p)  │
   │  (mcts.py +    │                                │       + (z-v)²   │
   │  self_play.py) │ ◄── network forward ─────┐     │       + L2       │
   └───────┬────────┘                          │     └────────┬─────────┘
           │ samples + outcome z               │              │ grad
           ▼                                   │              ▼
   ┌────────────────┐               ┌──────────┴──────┐ ┌──────────────┐
   │ z stamped      │               │  AZ network     │ │  Adam        │
   │ on every step  │               │  (network.py)   │◄┤  update      │
   └────────────────┘               │  policy + value │ └──────────────┘
                                    └─────────────────┘
```

### One self-play game (`self_play.py`)

```python
while game not over:
    counts = mcts.run(board)               # 64 simulations from current state
    π = visit_counts_to_policy(counts, τ)  # τ=1 first 12 plies, τ=0 thereafter
    record (state, π, side)
    sample action ~ π
    board.apply_move(...)

z = +1 / -1 / 0 from each side's POV (filled in after game ends)
push every (state, π, z) into ReplayBuffer
```

### One MCTS simulation (`mcts.py`)

From the root, repeatedly:

1. **Select** child action `a` maximizing `PUCT = -Q(child) + c_puct · P(s,a) · √(ΣN) / (1 + N(s,a))`.
   The minus on `Q` is the two-player flip — what's good for the child is bad for the parent.
2. **Apply** the move on a cloned board (the real board is never mutated).
3. **At a leaf:** call the network once on the leaf state. Mask illegal
   actions, renormalize policy, get value `v`.
4. **Back up** along the visited path. At each step up the tree, flip
   the sign of the value before adding it to that node's running sum.

After all simulations, the root's child visit counts are the move-selection
distribution. Counts are also the *training target* for the policy head —
"play in proportion to how much MCTS thought about each move."

### Training (`train.py`)

Each game contributes ≈ `ply_count` samples. After each game we run a
small number of Adam steps (`--updates-per-game`, default 8) on a random
batch from the buffer. The buffer is circular, persisted to disk on
every `--save-every` games, and reloaded on `--resume`.

```
loss = − Σ π_mcts · log softmax(p_logits)     # cross-entropy policy loss
       + (z − tanh(v_logit))²                  # MSE value loss
       + L2(weights)                           # built into layer kernels
```

## Files

| File | What it does |
|---|---|
| `encoder.py` | Board → `(5, 5, 7)` tensor in agent-POV coordinates; flat 56-action index ↔ `Move` object; legal-action mask. Same channel layout / action mapping as `quoridor_ai_move`. |
| `network.py` | Dual-head ResNet. Conv stem → N residual blocks (default 32 filters × 3 blocks, ≈ 80k params) → policy head (logits over 56 actions) + value head (scalar `tanh ∈ [-1, 1]`). Smaller than the DQN net because the 5×5 state space doesn't need more capacity. |
| `mcts.py` | PUCT MCTS. Mask-aware priors, two-player minimax backup, optional Dirichlet noise at the root for self-play exploration. Uses `QuoridorBoard.copy()` to simulate without mutating the real board. |
| `self_play.py` | One full game with MCTS for both sides. Returns `[(state, π_mcts, side)]` plus the outcome status; `z` is filled in for every sample once the game ends. |
| `replay_buffer.py` | Fixed-capacity circular buffer of `(s, π, z)` triples. Allocation-free sampling. Pickle save/load so a long run survives restarts. |
| `train.py` | The training loop above. Atomic save with 3-deep rotating backup (`az_net.weights.bak1.h5` … `bak3.h5`) so a bad run can be rolled back. |
| `ai_move_node.py` | ROS2 inference node. Subscribes to `/quoridor/compute_move_request`, runs MCTS with the trained network, publishes the chosen move. Wire-compatible with `quoridor_ai_move/ai_move_node`. |
| `launch/play_vs_az.launch.py` | Brings up `state_manager`, `web_interface`, and the AZ inference node. |

## Usage

### Build
```bash
cd /rs2_ws
colcon build --packages-select quoridor_alphazero --symlink-install
source install/setup.bash
```

### Train (fresh)
```bash
python3 -m quoridor_alphazero.train \
    --episodes 5000 --simulations 64 --updates-per-game 8 \
    --model-dir /rs2_ws/quoridor_az_models/run1 \
    --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1
```

### Train (resume)
Auto-loads weights, replay buffer, and switches to `--lr-resume` (2e-4):
```bash
python3 -m quoridor_alphazero.train --resume --episodes 5000 --model-dir /rs2_ws/quoridor_az_models/run1 --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1_2 --batch-size 1024 --lr-resume 8e-4

python3 -m quoridor_alphazero.train --resume --episodes 5000 --model-dir /rs2_ws/quoridor_az_models/run1 --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1_3 --batch-size 1024 --lr-resume 8e-4 --workers 12

```

### Play against it
You play through the web UI, the AZ engine plays bot. Inference can use
much heavier search than training — same weights, no retraining:
```bash
ros2 launch quoridor_alphazero play_vs_az.launch.py model_dir:=/rs2_ws/quoridor_az_models/run1 simulations:=400
```

### Recover from a bad checkpoint
Every save rotates the previous weights to `bak1.h5`, etc. To roll back:
```bash
cp /rs2_ws/quoridor_az_models/run1/az_net.weights.bak1.h5 \
   /rs2_ws/quoridor_az_models/run1/az_net.weights.h5
```

## Key hyperparameters

| Flag | Default | What it controls |
|---|---|---|
| `--simulations` | 64 | MCTS sims per move during self-play. More = stronger data but slower. At inference (ROS node) push to 200–400. |
| `--c-puct` | 1.5 | PUCT exploration weight. Higher = more exploration of low-prior moves. |
| `--dirichlet-eps` | 0.25 | Fraction of Dirichlet noise mixed into root priors during self-play (0 at inference). |
| `--temp-moves` | 12 | Plies of high-temperature (τ=1) move selection at the start of each game. After this point, plays argmax for sharper data. |
| `--updates-per-game` | 8 | Gradient steps after each self-play game. Bounded — can't run away on long games. |
| `--lr` / `--lr-resume` | 1e-3 / 2e-4 | Learning rate; `lr-resume` is used automatically when `--resume` is set, to avoid Adam overwriting prior weights. |
| `--filters` / `--blocks` | 32 / 3 | Network width / depth. Defaults are intentionally smaller than the DQN net — 5×5 doesn't need more. |

## TensorBoard signals to watch

| Tag | Healthy range | What it tells you |
|---|---|---|
| `loss/policy` | drops 4 → 2.5 | Network is starting to agree with MCTS. Stuck near 4 (= ln 56) means uniform predictions, search isn't biting yet. |
| `loss/value` | drops 0.5 → 0.1 | Network is learning to predict outcomes. Rising = something is overfitting or the buffer is dominated by stale games. |
| `game/avg_length_50` | 15–35 | Decisive games. Pinned at `--max-plies` (75) means search is too weak to find a winning line — increase `--simulations`. |
| `game/bot_win_rate_50` | near 0.5 | In self-play, both seats use the same network. A persistent gap means there's a first-move advantage the policy can't yet equalize. |
| `game/decisive_rate_50` | → 1.0 | Fraction of recent games that didn't time out. Should approach 1 once the network can finish games. |
| `loss/total` | declining | Sum of policy + value + L2. The headline number. |

## Practical tips

1. **Start small.** First validate the loop with `--simulations 16 --episodes 100`. If `loss/policy` is dropping, scale up. If it isn't, something is wrong with encoding/MCTS.
2. **Search > network capacity.** If results are weak, double `--simulations` before doubling `--filters`. Stronger search produces better training data, which trains the same net to play better.
3. **Inference search ≫ training search.** Train at 64 sims/move, play at 400+. Same model, much stronger play.
4. **Don't mix DQN and AZ checkpoints.** They share the encoder but the network architectures and weight filenames differ (`qnet.weights.h5` vs `az_net.weights.h5`). Use separate `model_dir`s.
5. **Save the buffer.** A populated `replay_buffer.pkl` is worth as much as the weights — it represents thousands of MCTS simulations. Don't `rm -rf` the model dir.

## What's intentionally absent vs `quoridor_ai_move`

| Feature in DQN package | Absent here, because… |
|---|---|
| ε-greedy exploration | MCTS + root Dirichlet noise replaces it. |
| Reward shaping (PBRS) | The value target is the actual game outcome — no shaping needed. |
| Step penalty | Same — outcome-only training. Stalling is naturally penalized: a long unfinished game gives `z = 0` to both sides, so the network learns there's no reward in dragging things out. |
| Online learning in the ROS node | AZ learning needs full self-play games at training-time scale. Per-move callbacks aren't the right granularity. Train offline with `train.py`, deploy the weights, repeat. |
| `train_vs_minimax` curriculum | Not needed: the network bootstraps from MCTS using its own (initially random) value estimates, which is enough signal to escape the cold start. If you want a curriculum, the right way is to use the minimax engine as the *opponent* in `play_game` — easy to add later. |

# Distributed

  What it does                                                                                                                                                                                
                                                                                                                                                                                              
  Same self-play / mp.Pool / SGD loop as train.py, but the model + optimizer are built inside tf.distribute.MultiWorkerMirroredStrategy().scope() — exactly the pattern from                  
  quoridor_ai_move/train.py:623,651,686. Each PC plays its own self-play games into its own local replay buffer; optimizer.apply_gradients all-reduces gradients across PCs every step so they
   stay in sync. Only the chief (TF_CONFIG task index 0) writes TensorBoard / saves checkpoints / saves the buffer.                                                                           
                                                                                                                                                                                            
  Launch                                                                                                                                                                                      
                                                                                                                                                                                              
  ## PC 0 (chief)                                                                                                                                                                              
  TF_CONFIG='{"cluster":{"worker":["192.168.6.122:12345","192.168.5.199:12345"]},"task":{"type":"worker","index":0}}' python -m train_distributed --distributed --workers 12 --resume --episodes 5000 --model-dir ~/rs2_ws/quoridor_az_models/run1 --tb-log-dir ~/rs2_ws/quoridor_az_tensorboard/run1_4 --batch-size 1024 --lr-resume 8e-4                            
                                                                                                                                                                                              
  ## PC 1                                                                                                                                                                                      
  TF_CONFIG='{"cluster":{"worker":["192.168.6.122:12345","192.168.5.199:12345"]},"task":{"type":"worker","index":1}}' python -m quoridor_alphazero.train_distributed --distributed --workers 12 --resume --episodes 5000 --model-dir ~/mnt/samba/quoridor_az_models/run1 --tb-log-dir ~/mnt/samba/quoridor_az_tensorboard/run1_4_1 --batch-size 1024 --lr-resume 8e-4                                                                                                       
   
  Without --distributed it falls back to tf.distribute.get_strategy() (or MirroredStrategy if you have multiple local GPUs) and behaves like train.py.                                        
                  
  Two things to know                                                                                                                                                                          
                  
  1. Effective batch scales with PC count. Per-PC batch is --batch-size; global batch is batch_size × num_workers. So with 2 PCs at --batch-size 1024 you're effectively training on 2048     
  samples per step. You may want to scale --lr with the global batch (or just let the larger batch act as variance reduction).
  2. Buffer-size barrier deadlock risk. Training only runs when the local buffer has ≥ batch_size. If one PC reaches that threshold and another hasn't, the cluster will deadlock on the      
  all-reduce. With identical configs (same --workers, --simulations, etc.) all PCs fill at the same rate, so this is fine in practice — but if you start one PC late or with very different   
  settings, you'll hang on the first SGD step. Fix is to either start them simultaneously or pre-load a shared replay buffer via --resume.
                                                                                                                                                                                              
  Run colcon build --packages-select quoridor_alphazero (or just pip install -e . since it's an egg-link install) before the new entry point is on PATH.
