Project Overview

QuorZero is the artificial intelligence (AI) component of a larger UR3e (Universal Robots e-Series, 3 kg payload) robotic arm system that plays Quoridor, a two-player strategy board game, against a human opponent. This portfolio focuses on the quoridor_alphazero Robot Operating System 2 (ROS 2) package: the neural network, the search algorithm, and the training pipeline.

Quoridor is a perfect-information two-player game with a branching factor that rewards strong planning. We replaced an earlier Deep Q-Network (DQN) baseline with Google's AlphaZero-style approach: a single network with a policy head and a value head, trained on data produced by Monte Carlo Tree Search that uses the same network as the leaf evaluator. The network initiates a search that is itself guided by the network. This configuration, unlike DQN, handles the two-player setting natively.

Approach

Why AlphaZero, not DQN?

DQN was designed for single-agent environments. In a two-player game, the Bellman bootstrap r + γ·max Q(s', a) confuses "my move quality" with "my opponent's reply quality" because s' is the state after the opponent moves. This produces reoccuring issues in self-play such as mutual stalling, oscillating policies, overfitted learning.

AlphaZero handles two players natively:

  • The value head is trained against the actual game outcome z ∈ {−1, 0, +1}.
  • The policy head is trained against the MCTS visit-count distribution, a strictly stronger signal than the network's own best guess.
  • Dual side learning in MCTS flips the sign of value at every level. What's good for the child is bad for the parent, helping it to learn how to attack and defend.

The self-play loop

Each self-play game uses MCTS for both sides, with the AlphaZero variant of tree search known as PUCT (Predictor + Upper-Confidence Tree). After every move the visit count distribution π is recorded alongside the state. When the game ends, the outcome z is stamped back onto every sample, and the triples (s, π, z) are pushed into a circular replay buffer. A small number of Adam (Adaptive Moment Estimation) optimiser steps follow each game. 

while game not over:
    counts = mcts.run(board)
    π = visit_counts_to_policy(counts, τ)
    record (state, π, side)
    sample action ~ π
    board.apply_move(...)

z = ±1 / 0 from each side's point of view
push every (state, π, z) into ReplayBuffer
loss = − Σ π_mcts · log softmax(p_logits)   # policy cross-entropy
       + (z − tanh(v_logit))²                 # value mean-squared error
       + L2(weights)                          # L2 weight decay

Training objective

The policy head is trained to imitate MCTS's visit count distribution over moves, and the value head is trained to predict the eventual game outcome z ∈ {-1, 0, +1}. Because the same network is the evaluator inside MCTS, every training step makes the search sharper, which produces better data for the next step. This results in the network continously getting better.

Build

cd /rs2_ws
colcon build --packages-select quoridor_alphazero --symlink-install
source install/setup.bash

Train from scratch

python3 -m quoridor_alphazero.train \
    --episodes 5000 --simulations 64 --updates-per-game 8 \
    --model-dir /rs2_ws/quoridor_az_models/run1 \
    --tb-log-dir /rs2_ws/quoridor_az_tensorboard/run1

Play against QuorZero

ros2 launch quoridor_alphazero play_vs_az.launch.py \
    model_dir:=/rs2_ws/quoridor_az_models/run1 \
    simulations:=400

Robotic System

The physical layer of the Quoridor robot — from camera pixels to gripper contact.

  • Perception
  • Robot Motion Control

The perception subsystem uses RGB and depth video from an Intel RealSense D435i depth camera mounted eye-in-hand on the UR3e end-effector. It produces a live digital board state — pawn occupancy on a 5×5 grid and wall placement on a 4×4 grid — along with 3D world coordinates for every detected piece, which the motion planning layer consumes directly.

Node Architecture

camera_node

Captures RGB and depth streams at 640×480 @ 15 FPS. Supports both live camera input and pre-recorded ROS bag playback for repeatable testing. Saves camera intrinsics to camera_intrinsics.json on first run and exposes /get_pawns and /get_walls ROS 2 services.

perception_node

Core detection node. Applies a homography transform to produce a fixed 500×500 top-down view, then detects pawns via binary thresholding on the warped image and walls via HSV red-colour segmentation. Publishes board state, wall state, 3D coordinates, and both AR and top-down visualisation streams at 10 Hz.

coordinate_node

One-shot calibration node. Detects all 25 pawn grid positions and 16 wall circle positions, then uses 8 hand-measured corner poses to bilinearly interpolate calibrated 3D world coordinates for every cell. Writes results to pawn_coords.txt and wall_coords.txt and shuts down.

Detection Pipeline

Board localisation uses Canny edge detection with CLAHE contrast normalisation, followed by contour finding and a 4-point homography warp. This produces a lighting-invariant top-down view regardless of camera angle.

Pawn detection applies binary thresholding on the warped grayscale image. Each of the 25 grid cells is sampled; a cell is marked occupied when >5% of its central ROI is dark.

Wall detection segments red blobs in HSV space using a two-range mask (hue wraps at 180°), then classifies each blob as horizontal or vertical based on its bounding-box aspect ratio.

3D localisation back-projects detected pixel centres through the RealSense depth frame using the camera intrinsics to produce (x, y, z) in the robot's coordinate frame, with ~0–2 cm accuracy after grid-based correction.

Published Topics

/perception/board_state

Int32MultiArray — 5×5

Flattened pawn occupancy grid. 1 = occupied, 0 = empty. Consumed by game logic and the control node's board-detection check.

/perception/wall_state

Int32MultiArray — 4×4

Wall placement and orientation. 0 = empty, 1 = vertical wall, 2 = horizontal wall.

/perception/pawns_3d & /perception/walls_inside_3d

Float32MultiArray

3D world coordinates (row, col, x, y, z) for every detected pawn and on-board wall. Lookup values come from the calibrated coordinate files.

/perception/augmented_reality & /perception/topdown

sensor_msgs/Image

Live camera feed with AR overlays (bounding boxes, orientation markers, board outline) and the warped 500×500 top-down view — both streamed at 10 Hz for real-time monitoring.
